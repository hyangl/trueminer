<blockquote>
<p>Ethereum miner with OpenCL, CUDA and stratum support</p>
</blockquote>
<p><strong>Ethminer</strong> is an Ethash GPU mining worker: with ethminer you can mine every coin which relies on an Ethash Proof of Work thus including Ethereum, Ethereum Classic, Metaverse, Musicoin, Ellaism, Pirl, Expanse and others. This is the actively maintained version of ethminer. It originates from <a href="https://github.com/ethereum/cpp-ethereum">cpp-ethereum</a> project (where GPU mining has been discontinued) and builds on the improvements made in <a href="https://github.com/Genoil/cpp-ethereum">Genoil's fork</a>. See <a href="#faq">FAQ</a> for more details.</p>
<h2><a id="user-content-features" class="anchor" aria-hidden="true" href="#features"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Features</h2>
<ul>
<li>OpenCL mining</li>
<li>Nvidia CUDA mining</li>
<li>realistic benchmarking against arbitrary epoch/DAG/blocknumber</li>
<li>on-GPU DAG generation (no more DAG files on disk)</li>
<li>stratum mining without proxy</li>
<li>OpenCL devices picking</li>
<li>farm failover (getwork + stratum)</li>
</ul>

<h3><a id="user-content-why-is-my-hashrate-with-nvidia-cards-on-windows-10-so-low" class="anchor" aria-hidden="true" href="#why-is-my-hashrate-with-nvidia-cards-on-windows-10-so-low"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Why is my hashrate with Nvidia cards on Windows 10 so low?</h3>
<p>The new WDDM 2.x driver on Windows 10 uses a different way of addressing the GPU. This is good for a lot of things, but not for ETH mining.</p>
<ul>
<li>For Kepler GPUs: I actually don't know. Please let me know what works best for good old Kepler.</li>
<li>For Maxwell 1 GPUs: Unfortunately the issue is a bit more serious on the GTX750Ti, already causing suboptimal performance on Win7 and Linux. Apparently about 4MH/s can still be reached on Linux, which, depending on ETH price, could still be profitable, considering the relatively low power draw.</li>
<li>For Maxwell 2 GPUs: There is a way of mining ETH at Win7/8/Linux speeds on Win10, by downgrading the GPU driver to a Win7 one (350.12 recommended) and using a build that was created using CUDA 6.5.</li>
<li>For Pascal GPUs: You have to use the latest WDDM 2.1 compatible drivers in combination with Windows 10 Anniversary edition in order to get the full potential of your Pascal GPU.</li>
</ul>
<h3><a id="user-content-why-is-a-gtx-1080-slower-than-a-gtx-1070" class="anchor" aria-hidden="true" href="#why-is-a-gtx-1080-slower-than-a-gtx-1070"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Why is a GTX 1080 slower than a GTX 1070?</h3>
<p>Because of the GDDR5X memory, which can't be fully utilized for ETH mining (yet).</p>
<h3><a id="user-content-are-amd-cards-also-affected-by-slowdowns-with-increasing-dag-size" class="anchor" aria-hidden="true" href="#are-amd-cards-also-affected-by-slowdowns-with-increasing-dag-size"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Are AMD cards also affected by slowdowns with increasing DAG size?</h3>
<p>Only GCN 1.0 GPUs (78x0, 79x0, 270, 280), but in a different way. You'll see that on each new epoch (30K blocks), the hashrate will go down a little bit.</p>
<h3><a id="user-content-can-i-still-mine-eth-with-my-2gb-gpu" class="anchor" aria-hidden="true" href="#can-i-still-mine-eth-with-my-2gb-gpu"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Can I still mine ETH with my 2GB GPU?</h3>
<p>Not really, your VRAM must be above the DAG size (Currently about 2.15 GB.) to get best performance. Without it severe hash loss will occur.</p>
<h3><a id="user-content-what-are-the-optimal-launch-parameters" class="anchor" aria-hidden="true" href="#what-are-the-optimal-launch-parameters"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>What are the optimal launch parameters?</h3>
<p>The default parameters are fine in most scenario's (CUDA). For OpenCL it varies a bit more. Just play around with the numbers and use powers of 2. GPU's like powers of 2.</p>
<h3><a id="user-content-what-does-the---cuda-parallel-hash-flag-do" class="anchor" aria-hidden="true" href="#what-does-the---cuda-parallel-hash-flag-do"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>What does the <code>--cuda-parallel-hash</code> flag do?</h3>
<p><a href="https://github.com/davilizh">@davilizh</a> made improvements to the CUDA kernel hashing process and added this flag to allow changing the number of tasks it runs in parallel. These improvements were optimised for GTX 1060 GPUs which saw a large increase in hashrate, GTX 1070 and GTX 1080/Ti GPUs saw some, but less, improvement. The default value is 4 (which does not need to be set with the flag) and in most cases this will provide the best performance.</p>
<h3><a id="user-content-what-is-ethminers-relationship-with-genoils-fork" class="anchor" aria-hidden="true" href="#what-is-ethminers-relationship-with-genoils-fork"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>What is ethminer's relationship with <a href="https://github.com/Genoil/cpp-ethereum">Genoil's fork</a>?</h3>
<p><a href="https://github.com/Genoil/cpp-ethereum">Genoil's fork</a> was the original source of this version, but as Genoil is no longer consistently maintaining that fork it became almost impossible for developers to get new code merged there. In the interests of progressing development without waiting for reviews this fork should be considered the active one and Genoil's as legacy code.</p>

<p>No, use geth, the go program made for ethereum by ethereum.</p>
<h3><a id="user-content-cuda-gpu-order-changes-sometimes-what-can-i-do" class="anchor" aria-hidden="true" href="#cuda-gpu-order-changes-sometimes-what-can-i-do"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>CUDA GPU order changes sometimes. What can I do?</h3>
<p>There is an environment var <code>CUDA_DEVICE_ORDER</code> which tells the Nvidia CUDA driver how to enumerates the graphic cards.
The following values are valid:</p>
<ul>
<li><code>FASTEST_FIRST</code> (Default) - causes CUDA to guess which device is fastest using a simple heuristic.</li>
<li><code>PCI_BUS_ID</code> - orders devices by PCI bus ID in ascending order.</li>
</ul>
<p>To prevent some unwanted changes in the order of your CUDA devices you <strong>might set the environment variable to <code>PCI_BUS_ID</code></strong>.
This can be done with one of the 2 ways:</p>
<ul>
<li>
<p>Linux:</p>
<ul>
<li>Adapt the <code>/etc/environment</code> file and add a line <code>CUDA_DEVICE_ORDER=PCI_BUS_ID</code></li>
<li>Adapt your start script launching ethminer and add a line <code>export CUDA_DEVICE_ORDER=PCI_BUS_ID</code></li>
</ul>
</li>
<li>
<p>Windows:</p>
<ul>
<li>Adapt your environment using the control panel (just search <code>setting environment windows control panel</code> using your favorite search engine)</li>
<li>Adapt your start (.bat) file launching ethminer and add a line <code>set CUDA_DEVICE_ORDER=PCI_BUS_ID</code> or <code>setx CUDA_DEVICE_ORDER PCI_BUS_ID</code>. For more info about <code>set</code> see <a href="https://docs.microsoft.com/en-us/windows-server/administration/windows-commands/set_1" rel="nofollow">here</a>, for more info about <code>setx</code> see <a href="https://docs.microsoft.com/en-us/windows-server/administration/windows-commands/setx" rel="nofollow">here</a></li>
</ul>
</li>
</ul>
